# 试点与落地实施指南

## 总体策略

采用“规范先对齐、单页先试点、验证后推广”的方式推进。

## 实施阶段

## 推进路径图

```mermaid
flowchart LR
    A[阶段 0 统一规范] --> B[阶段 1 单页试点]
    B --> C[阶段 2 项目内扩展]
    C --> D[阶段 3 团队推广]
```

### 阶段 0：统一规范

目标：

- 确认工件体系
- 确认角色边界
- 确认 AI 使用规则

输出：

- 正式文档体系

### 阶段 1：单页试点

目标：

- 选取一个真实页面跑通闭环

推荐页面：

- 后台列表页
- 详情页
- 中等复杂度表单页

输出：

- 完整工件链路
- review 结论
- 资产候选

推荐试点组织：

- 产品 1 人
- 设计 1 人
- 前端 1 人
- 交付负责人 1 人
- AI 执行者按同一套规范参与

### 阶段 2：项目内扩展

目标：

- 在同一项目内扩展到多个页面
- 收敛页面模式和 section schema

### 阶段 3：团队推广

目标：

- 固化模板
- 固化校验规则
- 建立资产目录

## 试点成功标准

- 能跑通完整闭环
- 工件可追溯
- review 可对照
- 至少沉淀 1 项可复用资产

建议同时记录以下指标：

- 从需求输入到 `Feature Brief` 的时间
- 从 `Design Contract` 到 `Page Spec` 的时间
- 从 `Page Spec` 到首版实现的时间
- 首轮 review 问题数量
- AI 输出直接采用比例
- 本轮新增资产数量

## 试点后要做的事

1. 回顾工件是否过重或缺失
2. 回顾哪些步骤适合工具化
3. 更新模板和资产目录

## 第一轮工具化优先级

试点后优先考虑以下工程化项：

1. 模板生成
2. 工件必填项检查
3. `Page Spec` 结构校验
4. review checklist 自动生成
5. `Implementation Record` 回写辅助

## 第一轮试点建议节奏

| 周期 | 重点动作 | 主要输出 |
| --- | --- | --- |
| 第 1 周 | 对齐页面与 owner | 页面信息卡、原始需求 |
| 第 2 周 | 完成三份核心上游工件 | `Feature Brief`、`Design Contract`、`Page Spec` |
| 第 3 周 | 实现与 review | 页面实现、`Review Checklist` |
| 第 4 周 | 回写与复盘 | `Implementation Record`、资产候选、工具化清单 |

## 常见失败方式

- 一上来试图全公司一次性推开
- 只谈理念，不跑真实页面
- 试点结束后不回写规范和资产

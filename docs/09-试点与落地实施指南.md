# 试点与落地实施指南

## 总体策略

采用“先跑真实页面、再收敛协议、最后扩展资产”的方式推进。

重点不是先做全套平台，而是先验证这套工程系统是否成立：

- 任意需求能否进入统一输入
- 任意 AI 执行器能否稳定接入
- 当前规则、当前规格和实现能否对齐
- 交付后能否产出资产候选

## 推荐推进路径

```mermaid
flowchart LR
    A[阶段 0 对齐体系] --> B[阶段 1 单页试点]
    B --> C[阶段 2 项目内扩展]
    C --> D[阶段 3 资产消费与推广]
```

## 阶段 0：对齐体系

目标：

- 确认统一输入与三种模式
- 确认责任边界与确认机制
- 确认 AI 执行器接入原则
- 确认回写与资产升级规则

输出：

- 正式文档体系
- 试点页面清单
- 试点度量口径

## 阶段 1：单页试点

目标：

- 选一个真实页面跑通闭环
- 验证标准模式或轻量模式是否真的可行

推荐页面：

- 后台列表页
- 详情页
- 中等复杂度表单页

推荐试点配置：

- 需求确认人 1 人
- 页面规则确认人 1 人
- 实现与回写负责人 1 人
- 交付裁决人 1 人
- 至少 1 个 AI 执行器按同一套协议参与

同一人可以兼任多个责任位。

输出：

- 页面闭环结果
- review 结论
- 资产候选
- 需要工具化的痛点清单

## 阶段 2：项目内扩展

目标：

- 在同一项目内扩展到多个页面
- 收敛页面模式和行为规格写法
- 验证 `Page Spec patch` 是否足够支撑小需求变更

## 阶段 3：资产消费与推广

目标：

- 固化模板、schema 和校验规则
- 建立资产目录与版本规则
- 让新需求能默认拿到历史资产

## 试点成功标准

- 能跑通完整闭环
- 当前规则、当前规格和实现结果可追溯
- review 可对照
- 至少沉淀 1 项可复用资产
- 第二个相似需求能消费第一轮资产中的至少 1 项内容

建议同时记录以下指标：

- 从原始输入到首个可执行规格的时间
- 从规格到首版实现的时间
- 首轮 review 问题数量
- AI 输出直接采用比例
- `Page Spec` / patch 同步及时率
- 本轮新增资产数量

## 第一轮工具化优先级

试点后优先考虑以下工程化项：

1. 原始输入标准化
2. 最小任务规格生成
3. `Page Spec` 结构校验与 patch 生成
4. review checklist 自动生成
5. `Implementation Record` 回写辅助
6. 资产候选提取与登记

## 第一轮建议节奏

| 周期 | 重点动作 | 主要输出 |
| --- | --- | --- |
| 第 1 周 | 对齐页面、模式与确认人 | 页面信息卡、原始输入 |
| 第 2 周 | 形成当前规则与当前规格 | `Feature Brief`、`Design Contract`、`Page Spec` 或 patch |
| 第 3 周 | 实现与 review | 页面实现、`Review Checklist` |
| 第 4 周 | 回写与复盘 | `Implementation Record`、资产候选、工具化清单 |

## 常见失败方式

- 一上来试图把所有页面都纳入标准模式
- 只谈理念，不跑真实页面
- 试点结束后不回写规范和资产
- 资产登记了，但没有设计后续消费方式

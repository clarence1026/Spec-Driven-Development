# 试点执行与成功标准

## 试点定位

这份文档聚焦：这套 AI 工程化系统怎样被验证，而不是停留在理念层。

它重点回答：

1. 为什么必须先做试点
2. 试点应该怎么选
3. 试点应该验证什么
4. 什么才算试点成功
5. 如何用试点支撑后续投入判断

## 为什么必须先试点

AI 工程化系统不能靠理念完整来证明价值，也不能靠单次 demo 来证明有效。

试点的作用是验证：

- 输入能否被稳定收敛
- 共享工件能否稳定驱动实现和 review
- AI 是否能在受控边界内真正参与
- 回写和资产沉淀是否能形成闭环

## 试点的真正短期目标

试点的短期目标不是证明“AI 能不能生成页面”，而是证明：

- 项目是否能稳定产出共享工件
- 共享工件是否能驱动实现和 review
- 每个项目结束后是否真的能留下资产

也就是说，试点的短期核心目标是：

`从项目推进中稳定积累资产`

## 试点验证总图

```mermaid
flowchart LR
    A[选定真实页面] --> B[按工件体系执行]
    B --> C[完成人机协同交付]
    C --> D[检查闭环是否成立]
    D --> E[评估指标与价值]
    E --> F[决定是否继续扩展]
```

## 先选什么页面

优先选下面这类页面：

- 后台列表页
- 详情页
- 中等复杂度表单页

不要选：

- 横跨多个系统的大流程页面
- 高度实验性的页面
- 还没有清晰输入的页面

## 先决定采用哪种模式

- 标准模式：需要完整 `Task Context`、页面规则表达、`Page Spec`
- 轻量模式：允许使用最小任务规格或 `Page Spec patch`
- 变更模式：从 `Change Request` 开始，先判断影响层级

## 试点最小参与结构

至少明确 4 个责任位：

1. 需求确认人
2. 页面规则确认人
3. 实现与回写负责人
4. 交付裁决人

如果团队已有 AI 执行器接入，还应明确：

5. AI 使用负责人或工作流维护人

## 试点执行顺序

1. 选一个真实页面
2. 判断采用哪种模式
3. 整理原始输入
4. 形成当前任务上下文、页面规则和页面规格
5. 基于规格实现页面
6. 做一轮 review
7. 回写 `Implementation Record`
8. 判断并登记资产候选
9. 记录本轮指标和问题

## 试点落地的 3 个执行抓手

如果要把试点从“原则”变成“运营动作”，建议固定抓住下面 3 个抓手：

1. 用 `docs/14-项目准入与试点运营清单.md` 统一准入分级、固定协同节点和执行铁律
2. 用 `docs/15-试点台账与度量模板.md` 记录每个页面的闭环状态、指标和复盘结果
3. 用 `docs/16-资产分级与升级门槛.md` 判断本轮沉淀是停留在项目层，还是进入共享层 / 平台层

这 3 个抓手解决的是：

- 项目怎么进
- 过程怎么管
- 资产怎么升

## 试点成功标准

一个试点页跑通，至少意味着：

- 原始输入、规则、规格和实现结果能对齐
- 页面完成了一轮 review
- 偏差被记录下来
- 至少沉淀 1 项资产候选
- 下一次相似需求可以复用本轮至少 1 项沉淀
- 团队能明确说出 AI 在哪几步真正产生了价值

如果试点完成后仍然没有资产候选、没有复用入口、没有 AI 参与闭环，就说明这套系统还没有真正建立起来。

## 推荐度量项

| 指标 | 作用 |
| --- | --- |
| 输入收敛完整率 | 衡量原始输入是否能稳定进入 `Task Context` |
| 规格同步率 | 衡量行为变化是否同步到 `Page Spec` 或 patch |
| review 驳回原因分布 | 衡量系统问题主要发生在哪一层 |
| 实现偏差率 | 衡量实现与规则/规格的一致性 |
| AI 辅助覆盖率 | 衡量 AI 参与了多少关键节点 |
| 回写完成率 | 衡量闭环是否真的成立 |
| 候选资产产出率 | 衡量是否形成后续复用能力 |

## 试点汇报时建议怎么讲

试点汇报不要只说“页面做完了”，而要按下面顺序讲：

1. 本轮试点验证了哪几个系统假设
2. AI 参与了哪些节点，替代了哪些低质量重复劳动
3. 哪些环节仍然依赖人工裁决
4. 闭环是否真正成立：收敛、实现、review、回写、沉淀是否都完成
5. 下一轮应该优先补哪一层能力

## 试点汇报结构建议

| 汇报部分 | 要回答什么 |
| --- | --- |
| 背景 | 为什么选这个页面做试点 |
| 过程 | 本轮按哪种模式运行，哪些共享工件被产出 |
| AI 参与 | AI 实际参与了哪些节点，价值在哪里 |
| 结果 | 实现、review、回写是否闭环 |
| 指标 | 试点数据是否支持继续推进 |
| 下一步 | 是继续试点、补规则，还是进入资产化 |

## 推荐节奏

| 周期 | 重点动作 | 主要输出 |
| --- | --- | --- |
| 第 1 周 | 定页面、定模式、定责任人 | 页面信息卡、原始输入 |
| 第 2 周 | 形成共享工件 | `Task Context`、页面规则表达、`Page Spec` 或 patch |
| 第 3 周 | 实现与 review | 页面实现、评审清单、差异记录 |
| 第 4 周 | 回写与复盘 | `Implementation Record`、资产候选、试点指标 |

## 常见失败方式

- 一上来就试图覆盖所有页面
- 只谈理念，不跑真实页面
- 试点结束后不回写规范和资产
- AI 只参与代码生成，没有进入收敛和校验环节
- 没有指标，导致试点价值无法复盘

## 当前没有样例目录时怎么做

如果仓库当前没有附带可直接复制的样例目录，不影响试点启动。

直接按业务仓库或临时试点目录创建一份执行包即可，关键是共享工件齐备、编号清晰、可追溯。

## 一句话结论

试点的目标不是证明这套体系“看起来完整”，而是证明这套 AI 工程化系统能在真实任务中形成收敛、执行、校验、回写和沉淀的最小闭环。

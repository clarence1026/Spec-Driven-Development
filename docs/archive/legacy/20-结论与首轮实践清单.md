# 结论与首轮实践清单

## 最终结论

基于当前整套文档的判断，可以直接下结论：

### 结论 1：方向是对的

`UI -> Frontend` 的 AI 工程化是成立的，而且贴合未来场景。

它不是“让 AI 写前端”的局部优化，而是把 `UI / PRD / Frontend / AI` 之间原本分散、依赖人脑拼接的交付方式，重构成：

`Input -> Spec -> Code -> Review -> Writeback -> Asset`

的工程系统。

### 结论 2：短期目标和长期目标是清晰的

- 短期目标：用真实页面跑通闭环，并持续积累资产
- 长期目标：把资产升级成平台能力，支撑模板选择、组件选择和受控生成

### 结论 3：最容易落地的执行方式已经明确

最优实践不是让所有人都多写文档，而是：

- AI 先起草
- 人做确认
- 页面做最小单元
- 项目先验证
- 资产再升级

### 结论 4：现在可以开始试点，不需要继续停留在理论讨论

当前最合理的下一步不是继续抽象，而是直接跑首个 `P1` 页面试点。

## 现在不要再争论的 6 件事

为了避免继续空转，下面 6 件事建议直接定下来，不再反复讨论：

1. `UI -> Frontend` 做 AI 工程化，方向成立
2. 页面是当前阶段最小实践单位
3. Spec 默认由 AI 生成草稿，人审核
4. UI 不负责写工程 Spec，只负责确认页面规则
5. 页面执行包先放项目仓，稳定资产再升公共仓
6. 平台化是后续阶段，不是当前起点

## 首轮试点就按这个最小方案跑

### 试点对象

首轮只选 1 个 `P1` 页面，优先：

- 后台列表页
- 详情页
- 中等复杂表单页

不要选：

- 跨系统大流程
- 需求仍在剧烈变化的页面
- 没有明确 UI / PRD / FE 责任人的页面

### 固定角色

首轮至少明确 4 个角色：

| 角色 | 职责 |
| --- | --- |
| PRD / 需求责任人 | 确认目标、范围、验收口径 |
| UI 责任人 | 确认页面规则 |
| Frontend 责任人 | 审核 Spec、实现、回写 |
| 负责人 / 裁决人 | 接受偏差、判断资产升级 |

如果已有 AI 工作流维护者，再补第 5 个角色：

| 角色 | 职责 |
| --- | --- |
| AI 使用 / 工作流责任人 | 负责 AI 起草和流程接入 |

## 首轮试点必须产出的 6 个文件

建议直接固定为下面 6 个文件，不要再犹豫：

```text
01-task-context.md
02-ui-rule-card.md
03-page-rules.md
04-page-spec.yaml
05-review-checklist.md
06-implementation-record.md
```

这 6 个文件的意义分别是：

| 文件 | 用途 |
| --- | --- |
| `01-task-context.md` | 收敛任务目标、范围、约束 |
| `02-ui-rule-card.md` | 让 UI 确认结构、状态、交互、边界 |
| `03-page-rules.md` | 将页面规则升级成前端和 AI 可消费表达 |
| `04-page-spec.yaml` | 作为实现和 review 的主输入 |
| `05-review-checklist.md` | 对照规则和 Spec 做一致性检查 |
| `06-implementation-record.md` | 回写偏差、结论、资产候选 |

## 首轮试点按 7 天节奏推进

### Day 1：选页 + 定责任人

- 选 1 个 `P1` 页面
- 明确 PRD / UI / FE / approver
- 确定本轮只做这一个页面

### Day 2：AI 起草 `Task Context`

- 整理 PRD、Figma、历史页面、聊天记录
- AI 生成 `01-task-context.md`
- 由 PRD / FE 确认

### Day 3：AI 起草 UI 规则确认卡

- AI 生成 `02-ui-rule-card.md`
- UI 只做确认和修正

### Day 4：生成页面规则和 Spec

- 基于确认卡生成 `03-page-rules.md`
- AI 生成 `04-page-spec.yaml`
- FE 审核 Spec 是否可实现

### Day 5-6：实现与评审

- FE 基于 Spec 实现
- AI 对照 Spec 输出 review 差异
- 人判断哪些差异接受、哪些必须修

### Day 7：回写与资产判断

- 更新 `06-implementation-record.md`
- 记录是否形成资产候选
- 更新试点台账

## 首轮试点只看 5 个结果

不要一开始看过多指标，首轮只看下面 5 个结果：

1. 是否形成了完整 `Task Context`
2. UI 页面规则是否被结构化确认
3. 是否生成并审核了 `Page Spec`
4. 是否完成了 review 和回写
5. 是否至少产出 1 个资产候选

如果这 5 个结果都成立，就说明这套方法已经不是概念，而是已经可跑。

## 首轮试点后怎么判断是否继续

### 继续推进的信号

- 团队没有因为这套流程明显变慢
- AI 在起草、整理、对照、回写上确实减少了人工负担
- 页面 Spec 能被前端直接消费
- 回写后至少留下 1 类可复用资产

### 暂停调整的信号

- 页面选得太大，导致流程拖不动
- UI 规则长期无法确认
- Spec 字段过重，前端不愿维护
- 资产没有消费入口，只是堆目录

## 现在最推荐的仓库分层

当前就按这条规则执行，不要再犹豫：

### 项目仓

放页面执行包：

- `Task Context`
- UI 页面规则确认卡
- 页面规则表达
- `Page Spec`
- review 清单
- 回写记录

### 公共仓

放稳定资产：

- 通用模板
- 通用 pattern
- 通用 review rule
- AI prompt / workflow / checker
- 案例与反例

### 平台层

先不急着做产品化，只定义方向：

- registry
- 在线模板选择
- 资产组合
- 受控生成

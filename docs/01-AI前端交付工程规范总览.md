# AI 前端交付工程规范总览

## 这套文档到底在解决什么

这套文档要解决的，不是“谁来写哪份文档”，而是：

- 任意需求输入如何进入统一工程链路
- 任意 AI 执行器如何稳定参与拆解、实现、验证与回写
- 页面当前规则和当前规格如何保持一致
- 每次交付如何沉淀成下次可直接复用的资产

所以，这不是某个 AI 工具的使用手册，也不是一套传统角色分工说明，而是一套面向 `PRD -> UI -> Frontend` 的工程系统规范。

## 这套体系的四个稳定对象

后续所有流程、工件、校验和沉淀，都围绕这四个稳定对象展开：

1. `Raw Input`：原始输入包
2. `Current Rules`：当前页面规则表达
3. `Current Spec`：当前行为规格表达
4. `Implemented Reality`：当前实现与回写结果

可以直白理解成：

- 原始输入包：这次到底要做什么
- 当前页面规则：页面应该遵守什么规则
- 当前行为规格：页面到底该怎么表现
- 当前实现结果：最后到底做成了什么

## 常用术语说明

| 术语 | 中文理解 | 在体系中的位置 |
| --- | --- | --- |
| `Feature Brief` | 功能简报 | 原始输入的标准化表达 |
| `Design Contract` | 设计契约 | 页面规则的标准化表达 |
| `Page Spec` | 页面规格 | 页面行为规格的标准化表达 |
| `Implementation Record` | 实现记录 | 实现结果与偏差的回写表达 |
| `Review Checklist` | 评审清单 | 验证与核对表达 |

它们都很重要，但不应被理解为“每次都必须手工先写完的五份文档”。

更准确地说：

- 这些工件是标准形态
- 可以由 AI 执行器生成初稿
- 关键结论必须有人确认
- 是否完整显式出现，取决于任务复杂度

## 体系的四层结构

```mermaid
flowchart LR
    A[统一输入层] --> B[AI 执行层]
    B --> C[验证与回写层]
    C --> D[资产复用层]
```

### 1. 统一输入层

解决：

- 需求从哪里来
- 以什么形式进入系统
- 缺失信息如何补齐

### 2. AI 执行层

解决：

- AI 如何理解任务
- AI 如何拿代码上下文与历史资产
- AI 如何生成规格、实现和修改建议

### 3. 验证与回写层

解决：

- 结果如何验证不跑偏
- 规则与规格如何保持一致
- 偏差如何被记录

### 4. 资产复用层

解决：

- 什么值得沉淀
- 资产落在哪里
- 新任务如何自动拿到旧资产

## 三种执行模式

这套体系不是只有一条流程，而是至少包含三种模式：

### 标准模式

适用：

- 新页面
- 关键页面
- 复杂交互页面
- 多方需要对齐的页面

典型链路：

`需求输入 -> Feature Brief -> Design Contract -> Page Spec -> 实现 -> Review -> Implementation Record -> 资产沉淀`

### 轻量模式

适用：

- 小需求
- 成熟模式复用
- 局部行为变更
- 已有规则和规格足够稳定的页面

典型链路：

`需求输入 -> 最小任务规格 / Page Spec patch -> 实现 -> Review -> Implementation Record -> 资产沉淀`

### 变更模式

适用：

- 已上线页面的持续变更
- 设计已改或规则已改但代码未同步
- 代码已改但规则/规格未回写

典型链路：

`Change Request -> 判断影响层级 -> 更新当前规则或当前规格 -> 实现 / 修正 -> Review -> 回写`

## 这套体系真正要统一的，不是工具，而是协议

这套体系不锁定任何单一 AI 工具。

真正需要稳定的是：

- 输入协议
- 页面规则表达协议
- 行为规格表达协议
- 验证协议
- 回写协议
- 资产升级与消费协议

AI 工具是可替换的，工程协议必须稳定。

## AI 执行器在体系里的位置

AI 执行器可以是：

- CLI 型工具
- IDE 内 agent
- 内部自动化流水线中的 agent
- 不同模型与技能组合

它在体系中的职责不是“代替人做决定”，而是：

- 读取统一输入
- 读取项目上下文和已有资产
- 生成或更新规则/规格表达
- 执行实现与基础验证
- 自动回写记录
- 提示资产候选

## 为什么这套体系值得做

如果没有这套体系：

- 每个需求都重新理解
- 每个页面都重新解释设计
- 每次 review 都重新争论标准
- AI 只能局部提效，无法形成稳定复利

如果这套体系成立：

- 任意需求都能进入统一输入
- 任意 AI 执行器都能接入统一流程
- 页面规则、行为规格与实现结果更容易保持一致
- 每次交付都能生成下一次任务可直接消费的资产

## 非目标

这套体系不解决：

- 业务方向本身是否正确
- 设计审美本身是否最佳
- 技术架构本身如何选型
- 组织结构本身如何分工

这套体系只解决一件事：

- 如何让前端交付在 AI 时代更稳、更快、更可复用

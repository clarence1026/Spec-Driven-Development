# 后台管理系统共性资产沉淀路线

## 目标定位

当前不建议一开始就抽象设计一套“大而全”的通用后台管理系统。

更稳妥的做法是：

`先基于成熟后台底座，从后台管理场景的高频页面切入，通过真实页面试点，逐步沉淀一套可复用的后台交付增量资产`

这套路线的目标不是一次性产出完整平台，也不是重复造一套后台 starter，而是先形成后台管理场景下可持续复用的：

- 页面模式
- 横切规则
- 执行资产

一句话边界：

`基础底座优先复用外部成熟 starter；内部只沉淀与业务交付、AI 协作强相关的增量资产`

## 为什么从后台开始

后台管理系统适合作为当前阶段的优先沉淀方向，主要有 4 个原因：

1. 页面形态重复度高，容易识别共性
2. 结构相对稳定，适合先沉淀规则和 Spec
3. 真实业务覆盖广，容易通过连续试点验证复用价值
4. 比营销页或强设计页面更适合先建立 AI 工程化闭环

因此，后台场景更适合作为当前公共仓的第一条重点资产线。

## 沉淀原则

后台管理系统资产沉淀建议遵循以下原则：

- 先复用成熟底座、后沉淀内部增量
- 先真实页面、后抽象体系
- 先高频模式、后特殊页面
- 先形成稳定规则、后讨论平台化消费
- 先完成一类页面闭环，再扩展下一类页面

一句话原则是：

`先从一个标准后台页面跑出闭环，再逐步升级成一套基于成熟底座的后台交付资产`

## 推荐分层

后台管理场景建议按 4 层组织沉淀对象：

### 1. 基础底座（优先外部复用）

用于承接后台页面的通用框架和工程能力。

这部分优先复用成熟 starter / 组件库，不作为第一批内部重点资产。

典型对象包括：

- layout / sidebar / header / breadcrumb
- table / form / modal / drawer / pagination
- router / request / auth 接入 / theme / i18n

当前策略是：

- 优先选型和复用
- 必要时做轻封装
- 不重复造通用后台框架

### 2. 页面模式

用于承接后台高频页面形态。

建议按以下顺序推进：

- `list-page-table`
- `form-page-basic`
- `detail-page-basic`
- `list-page-card`

### 3. 横切规则

用于承接不属于单一页面、但会反复出现在后台页面中的共性规则。

建议优先沉淀：

- `permission-rule`
- `responsive-admin-rule`
- `feedback-rule`
- `data-state-rule`

### 4. 执行资产

用于承接模板、Spec、review 和案例等执行层对象。

建议优先沉淀：

- template
- page spec
- review checklist
- prompt / workflow
- case

## 外部复用与内部沉淀边界

为避免重复建设，当前阶段建议明确以下边界：

| 类别 | 典型内容 | 处理方式 |
| --- | --- | --- |
| 基础框架与基础组件 | layout、table、form、drawer、pagination | 优先复用外部 starter / 组件库 |
| 工程基础设施 | router、request、auth 接入、theme、i18n | 优先复用或轻封装 |
| 页面模式 | `list-page-table`、`form-page-basic`、`detail-page-basic` | 内部沉淀 |
| 横切规则 | `permission-rule`、`responsive-admin-rule`、`data-state-rule` | 内部沉淀 |
| 执行资产 | `page-spec-template`、`review-admin-checklist`、prompt / workflow | 内部沉淀 |

这条边界的目的只有一个：

`不重复造成熟底座，只沉淀团队会反复使用的业务交付资产`

## 第一阶段范围

第一阶段建议定义为：

`基于成熟后台底座，沉淀后台管理场景的第一批增量交付资产`

首批覆盖方向建议包括：

- 列表页模式
- 权限规则
- 三端适配规则
- review 规则
- page spec 模板

但首轮试点不建议同时覆盖全部对象，而是先从一类最稳定、最高频的页面切入。

## 首轮试点建议

首轮试点建议选择：

`标准后台表格列表页`

典型页面包括：

- 用户列表
- 订单列表
- 内容列表

首轮默认覆盖范围建议如下：

- 页面形态：以表格列表为主
- 终端范围：`PC / Pad / Mobile` 三端全覆盖
- 页面能力：筛选、分页、表格列展示、行级操作、加载态、空态、错误态
- 资产目标：在完成单页面的同时，顺手抽出页面模式、横切规则和执行资产

首轮暂不强行纳入：

- 自建后台基础框架
- 卡片列表
- 复杂联动表单
- 审批流
- 高度定制业务页
- 特殊大盘页

## 首批建议沉淀的资产名

围绕首轮试点，建议优先沉淀以下对象：

- `list-page-table`
- `permission-rule`
- `responsive-admin-rule`
- `review-admin-checklist`
- `page-spec-template`

第二批可在首轮验证后再补：

- `form-page-basic`
- `detail-page-basic`
- `list-page-card`

## 三端适配建议

既然首轮要求三端全覆盖，建议在规划中明确不同终端的目标表达方式：

- `PC`：保留完整信息密度与完整操作能力
- `Pad`：保留主要结构与主要操作，压缩布局密度
- `Mobile`：保留核心筛选、核心字段与核心操作，采用堆叠、折叠或分步表达

这里的关键不是“能力缩水”，而是：

`同一业务能力在不同终端下采用不同表达形式`

## 权限规划建议

权限应作为首批横切规则纳入规划，但具体覆盖深度建议保留为会上待定项，不在当前阶段提前定死。

会上建议重点讨论：

- 是否只覆盖页面级和按钮级权限
- 是否纳入区块级权限
- 是否纳入字段 / 列级权限

这样既保证方向清晰，也避免首轮过度设计。

## 分阶段路线

### 阶段 1：标准表格列表页

目标：

- 跑通一个真实后台列表页闭环
- 初步沉淀 `list-page-table`、三端规则、权限口径和 review / spec 资产

### 阶段 2：基础表单与详情页

目标：

- 补齐 `form-page-basic` 与 `detail-page-basic`
- 让后台管理系统从“看列表”扩展到“查详情、做编辑”

### 阶段 3：扩展复杂模式

目标：

- 逐步纳入卡片列表、批量操作、复杂筛选、复杂权限等扩展能力
- 将验证稳定的对象升级为共享资产和平台消费对象

## 会上建议重点讨论的待定项

首轮规划建议明确以下待定项，供内部会议裁决：

1. 首轮试点具体选哪个列表页
2. 权限首轮覆盖到哪一层
3. 列表页首轮是否纳入批量操作
4. Mobile 端列表采用横向滚动还是字段裁剪
5. 第一批哪些对象进入共享资产，哪些继续留在项目级验证

## 成功信号

如果这条路线推进顺利，建议优先观察以下信号：

- 两个以上后台列表页开始复用相同页面模式
- 三端适配开始出现稳定规则而非临时 patch
- 页面级 review 开始使用统一 checklist
- 列表页 Spec 写法开始趋于稳定
- 第二批页面能够直接复用第一批沉淀资产

# 完整单页闭环案例

## 场景

以下示例演示一个后台“订单列表页”的完整闭环，并顺带说明小变更时如何使用 `Page Spec patch`。

## 一、原始输入包摘要

```json
{
  "title": "订单列表页优化",
  "goal": "帮助运营快速查看、筛选和处理订单",
  "scope": ["订单列表", "状态筛选", "时间筛选", "分页"],
  "references": {
    "prd": ["订单管理需求说明"],
    "figma": ["订单列表页设计稿链接"]
  },
  "constraints": ["复用现有表格组件", "保留现有权限模型"],
  "acceptanceHints": ["支持按状态和时间筛选", "状态完整"]
}
```

## 二、Feature Brief 摘要

```md
# Feature Brief

## Feature Name
Order List

## Goal
帮助运营快速查看、筛选和处理订单。

## Users
运营人员

## User Scenarios
1. 查看订单列表
2. 按状态和时间筛选订单
3. 查看订单详情
```

## 三、Design Contract 摘要

```md
# Design Contract

## Page
Order List

## Layout
顶部筛选区 + 中部表格区 + 底部分页区

## Components
- FilterBar
- OrderTable
- Pagination

## Page States
- loading
- empty
- error
- ready
```

## 四、Page Spec 摘要

```json
{
  "page": "order-list",
  "route": "/orders",
  "layout": "list-page",
  "permissions": ["orders.read"],
  "states": ["loading", "empty", "error", "ready"],
  "sections": [
    { "type": "filter-bar" },
    { "type": "data-table", "dataSource": "orders" },
    { "type": "pagination" }
  ]
}
```

## 五、实现与 Review

review 时重点核对：

- 页面结构是否符合当前页面规则
- section 与 `Page Spec` 是否一致
- loading / empty / error / ready 是否完整
- 筛选、详情和分页交互是否符合定义

## 六、Implementation Record 摘要

回写时至少应记录：

- 关键技术决策
- 偏差项
- review 证据
- 资产候选

例如：

- 复用了既有 `DataTable`
- 新增 `empty` 状态的统一提示组件
- 筛选项文案长度与设计稿略有调整，已记录偏差

## 七、小变更示例：新增“支付方式”筛选

如果后续只是新增一个筛选项，这通常不需要重写整份 `Page Spec`，但需要同步行为规格。

示例 patch：

```json
{
  "changeSummary": "新增支付方式筛选",
  "affectedSections": ["filter-bar"],
  "affectedStates": [],
  "affectedInteractions": ["filter-submit"],
  "decision": "更新 Page Spec patch",
  "updatedBy": "实现与回写负责人"
}
```

这次变更至少需要：

- 更新当前行为规格表达
- review 时对照 patch
- 在 `Implementation Record` 中记录这次调整

## 八、案例中的资产候选

本案例至少可沉淀以下资产：

- 后台列表页 `Page Spec` 写法
- `FilterBar + DataTable + Pagination` 组合模式
- 列表页 review checklist 默认项
- 列表页小变更的 `Page Spec patch` 写法

## 九、案例价值

这个案例的重点不是页面本身，而是演示：

- 业务如何进入统一输入
- 页面规则如何形成工程表达
- 页面如何形成当前行为规格
- review 和记录如何形成闭环
- 小变更如何通过 patch 保持规格同步
- 资产如何沉淀为下次项目的默认输入

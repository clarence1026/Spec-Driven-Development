# 完整单页闭环案例

## 场景

以下示例演示一个后台“订单列表页”的完整闭环。

## 一、Feature Brief 摘要

```md
# Feature Brief

## Feature Name
Order List

## Goal
帮助运营快速查看、筛选和处理订单。

## Users
运营人员

## User Scenarios
1. 查看订单列表
2. 按状态和时间筛选订单
3. 查看订单详情
```

## 二、Design Contract 摘要

```md
# Design Contract

## Page
Order List

## Layout
顶部筛选区 + 中部表格区 + 底部分页区

## Components
- FilterBar
- OrderTable
- Pagination

## Page States
- loading
- empty
- error
- ready
```

## 三、Page Spec 摘要

```json
{
  "page": "order-list",
  "route": "/orders",
  "layout": "list-page",
  "permissions": ["orders.read"],
  "states": ["loading", "empty", "error", "ready"],
  "sections": [
    { "type": "filter-bar" },
    { "type": "data-table", "dataSource": "orders" },
    { "type": "pagination" }
  ]
}
```

## 四、Review 与回写

review 时重点核对：

- 页面结构是否符合 `Design Contract`
- section 与 `Page Spec` 是否一致
- loading / empty / error / ready 是否完整
- 筛选、详情和分页交互是否符合定义

回写 `Implementation Record` 时，至少应记录：

- 关键技术决策
- 偏差项
- review 证据
- 资产候选

## 五、案例价值

这个案例的重点不是页面本身，而是演示：

- 业务如何进入结构化输入
- 设计如何形成工程契约
- 页面如何形成实现规格
- review 和记录如何形成闭环
- 资产如何沉淀为下次项目的默认输入

## 六、案例中的资产候选

本案例至少可沉淀以下资产：

- 后台列表页 `Page Spec` 写法
- `FilterBar + DataTable + Pagination` 组合模式
- 列表页 review checklist 默认项
